# 城市空气质量监测与预测系统设计文档

本文档从“目标-方案-权衡-演进”视角描述项目设计，覆盖当前落地实现，并为后续迭代提供统一参照。

## 1. 设计目标

### 1.1 业务目标

- 构建可面向全国、省、市多层级展示的空气质量可视化大屏。
- 提供近实时观测、历史趋势、空间分布、预警研判与预测联动能力。
- 支持“演示可用 + 工程可扩展”，兼顾展示效果与工程落地性。

### 1.2 技术目标

- 数据采集、接口服务、离线模型相互解耦，降低耦合风险。
- 统一 API 协议和数据口径，保证前端组件可复用、可组合。
- 支持多数据源回退与离线数据兜底，提升系统可用性。
- 保持模型产物协议稳定，便于平滑替换算法。

### 1.3 非功能目标

- 可运维：健康检查、调试接口、可观测日志。
- 可扩展：新增城市、指标、模型不破坏现有主链路。
- 可维护：目录职责清晰，文档可直接指导开发与部署。

## 2. 总体方案

### 2.1 分层架构方案

系统采用四层设计：

1) 数据接入层：Open-Meteo / QWeather / 聚合 API / CSV 导入  
2) 数据服务层：同步任务 + MySQL 持久化  
3) 接口层：Express 统一聚合查询与模型产物读取  
4) 展示层：Vue 大屏与多面板联动交互

### 2.2 关键模块方案

- 前端：`air-quality-visualization/frontend`
  - 主视图：`src/views/Dashboard.vue`、`src/views/PublicDisplayScreen.vue`
  - API 封装：`src/api.js`
  - 业务组件：地图、排行、趋势、预警、模型联动面板
- 后端：`air-quality-visualization/backend`
  - 接口服务：`src/server.js`
  - 定时同步：`src/index.js` + `src/syncJob.js`
  - 数据导入：`src/importWeatherCsv.js`
- 模型：`air-quality-visualization/model`
  - 树模型：`train_dashboard_model.py`
  - LSTM：`train_dashboard_lstm_model.py`
  - Transformer：`train_dashboard_transformer_model.py`
  - 极端天气因子脚本：`weather_extreme_predict_payload.py`

### 2.3 数据流方案

- 同步任务按固定频率拉取各城市最新观测并 upsert 入库。
- API 按“全国/省/市”作用域聚合，返回统一 JSON 协议。
- 模型脚本离线生成 payload，API 统一透出给前端模型面板。
- 前端基于同一城市上下文，实现多图联动和模型结果叠加。

## 3. 核心设计权衡

### 3.1 同步任务与 API 分离

- 选择：拆为两个进程。
- 收益：采集波动不影响 API 可用；API 重启不影响同步。
- 代价：部署与监控对象增多，需要进程管理策略。

### 3.2 历史表与最新表并存

- 选择：`city_observation_latest` + `city_observation_minute` 双表。
- 收益：最新值查询快，历史趋势可追溯。
- 代价：写入路径更复杂，需保证双写一致性。

### 3.3 离线模型产物驱动前端

- 选择：模型脚本输出 JSON/CSV，后端读取返回。
- 收益：前后端不耦合训练过程，便于切换模型版本。
- 代价：需要额外管理产物生命周期与版本一致性。

### 3.4 多数据源兼容与回退

- 选择：支持 Open-Meteo、QWeather、聚合 API、CSV 回填。
- 收益：外部接口异常时仍可运行。
- 代价：字段映射与异常处理逻辑更复杂。

### 3.5 历史粒度策略

- 现状：历史表命名为 `minute`，但当前按小时粒度对齐写入。
- 原因：兼容历史命名并降低存储与计算压力。
- 风险：命名与实际粒度存在认知偏差，需文档明确。

## 4. 详细设计

### 4.1 数据库设计

核心表：

- `city`：城市维度信息（adcode、name、province、经纬度）
- `city_observation_latest`：每城最新快照
- `city_observation_minute`：历史记录（当前小时粒度）

写入策略：

- 统一 `INSERT ... ON DUPLICATE KEY UPDATE`
- 最新值写 `latest`，历史值按对齐时间写 `minute`
- 避免 DELETE/TRUNCATE，保留完整追溯链路

### 4.2 接口设计

接口遵循统一响应：

- 成功：`{ ok: true, data: ... }`
- 失败：`{ ok: false, error: "..." }`

主要分组：

- 健康与调试：`/api/health`、`/api/debug/*`
- 地图与总览：`/api/map/nation`、`/api/map/province`、`/api/metrics/latest`
- 趋势分析：`/api/series/pm25`、`/api/weather/records`、`/api/weather/trend`、`/api/rank`
- 对比与研判：`/api/contrast/top`、`/api/contrast/detail`
- 模型联动：`/api/model/dashboard`、`/api/model/versions`、`/api/model/predict*`
- 极端天气：`/api/weather/extreme/hourly`

### 4.3 前端联动设计

- 统一请求入口：`frontend/src/api.js`
- 城市上下文统一解析：通过 `/api/city/resolve` 校正“全国/省/市”输入
- 页面结构：地图为中心，排行、趋势、预警、模型面板围绕同一上下文刷新

### 4.4 模型设计

当前模型体系并行支持：

- 树模型（XGBoost/GBDT）：高效稳定、训练速度快
- LSTM：建模时序依赖，支持多目标多步预测
- Transformer：强化长程依赖表达能力
- 规则模型：面向生活指数与极端天气场景，解释性强

模型输出统一沉淀在 `backend/data/model_outputs`，后端统一对外透出。

## 5. 演进路线

### 5.1 近期（1-2 个迭代）

- 增加模型产物自动刷新任务（定时训练或定时生成）
- 补充接口级缓存（省级/全国聚合）
- 增强模型看板中的版本对比与回滚能力

### 5.2 中期（3-5 个迭代）

- 引入任务编排（训练、产物校验、发布）
- 建立端到端质量基线（接口回归 + 前端可视回归）
- 建立异常告警（数据源失败、产物缺失、关键接口超时）

### 5.3 长期

- 形成“实时观测 + 预测推演 + 风险研判”闭环平台
- 引入更细粒度时空建模与区域传输影响分析
- 将模型管理升级为可观测、可评估、可审计的 MLOps 流程

## 6. 设计结论

当前系统在工程上已形成“数据采集-存储-服务-展示-预测”的完整闭环，具备上线演示与持续迭代基础。  
核心设计强调“解耦、容错、可替换、可扩展”，适合继续向高可用与智能化方向演进。
