# 运行手册（部署、运维、排障）

本文档面向开发、测试和运维，覆盖从零部署到问题排查的完整流程。

## 1. 系统总览

系统由三部分组成：

- 前端：`frontend`（Vue + Vite）
- 后端 API：`backend/src/server.js`（默认端口 `3001`）
- 数据同步任务：`backend/src/index.js`（每 5 分钟同步一次）

数据主流程：

1) 同步任务从外部数据源抓取天气/空气数据  
2) 入库到 `city_observation_minute`（历史）和 `city_observation_latest`（最新）  
3) API 聚合并输出给前端展示  
4) （可选）模型脚本输出 JSON，API 透传给前端模型面板

## 2. 部署前检查清单

- Node.js / npm 版本满足要求
- MySQL 服务可连接
- `backend/.env` 已配置
- 端口未被占用：
  - 后端 API：`3001`
  - 前端开发服务器：Vite 默认端口（通常 `5173`）
- 防火墙/安全组已放行（如跨机访问）

## 3. 首次部署步骤

### 3.1 数据库初始化

在 MySQL 执行：

- `backend/schema.sql`

确认表存在：

- `city`
- `city_observation_minute`
- `city_observation_latest`

### 3.2 配置后端环境变量

在 `backend/`：

```bash
cp .env.example .env
```

必须确认：

- MySQL 连接信息正确
- 数据源参数有效（Open-Meteo 或 QWeather）
- `CITY_SEED_LIST_PATH` 路径有效（默认 `./data/prefecture_cities_seed.json`）

### 3.3 安装依赖并冷启动

在 `backend/`：

```bash
npm i
npm run build:seed
npm run once
npm run server
```

`npm run once` 用于验证单次同步是否成功；通过后再启常驻同步。

### 3.4 启动常驻同步

在 `backend/`：

```bash
npm start
```

任务按 cron `*/5 * * * *` 每 5 分钟执行一次。

### 3.5 启动前端

在 `frontend/`：

```bash
npm i
npm run dev
```

若后端非本机地址，修改 `frontend/.env.local` 的 `VITE_API_BASE`。

## 4. 日常运行建议

- API 服务与同步任务建议分开进程运行，便于故障隔离
- 对 `backend` 日志做滚动切分（按日或按大小）
- MySQL 定期备份，尤其是 `city_observation_minute`
- 对历史表设置容量策略（按业务周期归档）

## 5. 常用运维命令

在 `backend/`：

- 仅同步一次：
  ```bash
  npm run once
  ```
- 启动 API：
  ```bash
  npm run server
  ```
- 启动定时同步：
  ```bash
  npm start
  ```
- CSV 导入（预演）：
  ```bash
  npm run import:weather-csv -- ./data/file.csv --dry-run
  ```
- CSV 导入（正式）：
  ```bash
  npm run import:weather-csv -- ./data/file.csv --update-latest
  ```

## 6. 健康检查与监控建议

建议纳入监控的接口：

- `GET /api/health`：数据库连接状态
- `GET /api/debug/counts`：数据量及最新时间戳
- `GET /api/debug/missing?staleMinutes=15`：缺失/过期城市诊断

建议设置告警阈值：

- `latest_max_ts` 超过 15~20 分钟未更新
- `no_latest_cnt`、`stale_cnt` 异常升高
- API 5xx 比例持续升高

## 7. 模型文件运行（可选）

在项目根目录执行：

```bash
python model/train_dashboard_model.py --csv ./weather_air_14days_full.csv
python model/weather_extreme_predict_payload.py --csv ./weather_air_14days_full.csv
```

输出目录默认：

- `backend/data/model_outputs/dashboard_payload.json`
- `backend/data/model_outputs/weather_extreme_hourly_payload.json`

若 API 提示模型文件不存在，优先检查：

- 文件路径是否正确
- 脚本执行是否成功
- 运行用户对目录是否有读写权限

## 8. 故障排查手册

### 8.1 前端空白或请求失败

排查顺序：

1) 后端 `npm run server` 是否已启动  
2) `VITE_API_BASE` 是否指向正确地址  
3) 浏览器控制台是否有跨域/网络错误  
4) `GET /api/health` 是否返回 `{ ok: true }`

### 8.2 后端启动报数据库连接错误

检查：

- `backend/.env` 的 `MYSQL_*` 是否正确
- MySQL 是否允许该用户从当前主机连接
- 数据库名是否与 `schema.sql` 一致

### 8.3 同步任务有日志但无数据入库

检查：

- 数据源配置是否正确（Open-Meteo/QWeather）
- `city` 表是否有种子数据（先执行 `npm run build:seed`）
- `GET /api/debug/counts` 中 minute/latest 计数是否增长

### 8.4 CSV 导入失败

检查：

- CSV 编码建议 UTF-8（可带 BOM）
- 时间列是否存在（`ts` / `time` / `datetime` / `时间`）
- 城市标识列是否可匹配（`adcode` 或 `name + province`）

### 8.5 模型面板无数据

检查：

- 模型输出 JSON 是否存在于 `backend/data/model_outputs/`
- 文件内容是否为合法 JSON，且 `ok=true`
- API 读取路径环境变量是否被改动

## 9. 生产环境建议

- 使用进程管理器（如 PM2、systemd）守护 `server` 与 `start`
- 使用反向代理统一入口（如 Nginx）
- 按环境分离配置（开发/测试/生产）
- 为数据库与模型输出目录设置备份和清理策略

## 10. 变更操作建议

每次升级建议按以下顺序：

1) 备份数据库  
2) 更新代码  
3) 安装依赖  
4) 执行迁移 SQL（如有）  
5) 先 `npm run once` 验证，再恢复常驻任务  
6) 回归检查核心接口与前端页面
