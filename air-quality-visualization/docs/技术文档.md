# 技术文档（架构、接口、模型与运维）

本文档面向开发、测试与运维人员，重点说明本项目的技术实现方式、关键接口、模型管线及上线维护要点。

## 1. 系统架构总览

项目采用前后端分离 + 离线模型产物联动架构。

- 前端：`frontend`（Vue 3 + Vite + ECharts）
- 后端：`backend`（Node.js + Express + MySQL）
- 模型：`model`（Python，树模型/LSTM/Transformer + 规则脚本）
- 文档：`docs`

核心数据流如下：

1) 同步任务从外部数据源拉取城市气象与空气数据。  
2) 数据落库至城市维表、最新表、历史表。  
3) 前端通过 API 获取地图、趋势、排行、天气与模型结果。  
4) 模型脚本生成 JSON/CSV 产物，由后端接口统一对外提供。

## 2. 后端实现

### 2.1 进程职责

- `npm run server`：启动 API 服务（`backend/src/server.js`）
- `npm start`：启动定时同步（`backend/src/index.js`）
- `npm run once`：执行一次同步（联调诊断）

该设计将“数据同步”和“接口服务”解耦，避免采集异常影响查询可用性。

### 2.2 核心模块

- `backend/src/syncJob.js`：统一数据拉取、清洗、映射、upsert 写库
- `backend/src/server.js`：REST 接口层，提供查询与模型产物读取
- `backend/src/importWeatherCsv.js`：离线 CSV 数据导入
- `backend/src/buildSeed.js`：城市种子构建与预处理

### 2.3 数据写入策略

- 采用 `INSERT ... ON DUPLICATE KEY UPDATE`
- `city_observation_latest`：每城市最新快照
- `city_observation_minute`：历史序列（当前业务按小时粒度对齐写入）
- 不做全表删除，保留历史可追溯性

## 3. 数据库设计

### 3.1 主要数据表

- `city`：城市基础信息（adcode、名称、省份、经纬度等）
- `city_observation_latest`：每城市最新观测
- `city_observation_minute`：历史观测序列

### 3.2 字段分层

- 气象因子：`temp_c`、`humidity`、`wind_speed`、`pressure_hpa`、`precip_mm`
- 空气质量：`aqi`、`pm25`、`pm10`、`no2`、`so2`、`o3`、`co`
- 元信息：`source`、`ts`

### 3.3 查询与索引建议

- 历史查询高频路径：`(city_id, ts)` 复合主键
- 省级聚合建议增加省份映射缓存
- 高频趋势接口可考虑按天/小时预聚合表

## 5. 前端实现

### 5.1 请求层

`frontend/src/api.js` 提供统一 `apiGet/apiPost` 能力，支持：

- `VITE_API_BASE` 指定服务地址
- 多后备地址容错（本地多端口切换）
- 统一错误处理与日志

### 5.2 可视化层

核心页面：`frontend/src/views/Dashboard.vue`、`frontend/src/views/PublicDisplayScreen.vue`

核心组件包含：

- 地图与区域联动（全国/省/市）
- 趋势与历史分析
- 排行与对比分析
- 生活气象与极端天气预警
- 模型版本、训练状态与预测结果展示

## 6. 模型与算法实现

### 6.1 树模型管线

文件：`model/train_dashboard_model.py`

- 通过时间拆分训练/测试
- 构建时序滞后与滚动特征
- 优先使用 XGBoost，缺失时回退 sklearn GBDT
- 多目标回归：AQI、PM2.5、PM10、NO2、O3
- 导出大屏统一 payload 与分析 CSV

### 6.2 LSTM 管线

文件：`model/train_dashboard_lstm_model.py`

- 城市 embedding + 多目标序列输出
- 未来多步直接预测（默认 168 小时）
- 输出模型权重、scaler 与 payload
- 通过梯度敏感度计算特征重要性

### 6.3 Transformer 管线

文件：`model/train_dashboard_transformer_model.py`

- 编码器结构 + 位置编码
- 多城市多目标多步预测
- 产物结构与 LSTM/树模型保持一致

### 6.4 规则模型

- `model/weather_extreme_predict_payload.py`：极端天气小时因子外推
- `frontend/src/models/lifeMeteorologicalModel.js`：生活气象指数规则计算

## 7. 运维与发布

### 7.1 环境变量

后端关键变量包括：

- MySQL 连接：`MYSQL_HOST`、`MYSQL_PORT`、`MYSQL_USER`、`MYSQL_PASSWORD`、`MYSQL_DATABASE`
- 数据源开关与参数：`OPEN_METEO_ENABLED`、`QWEATHER_KEY`、`API_URL`、`CITY_SEED_LIST_PATH`

前端变量：

- `VITE_API_BASE`

### 7.2 启动方式

- 推荐本地启动脚本：`一键启动-前后端.bat`
- 生产建议：
  - API 与同步任务分服务部署
  - 配置日志轮转与定时健康检查
  - 对模型产物目录做持久化与版本管理

### 7.3 日常运维检查

- `/api/health` 是否正常
- 同步任务是否持续写入最新表
- 模型产物文件时间戳是否更新
- 前端核心看板接口是否返回非空数据

## 8. 已知边界与后续优化

- 历史表名为 `minute`，当前实际策略为小时粒度
- 公共数据源可能限流，需具备回退与限流参数
- 模型结果依赖离线脚本执行，建议加入自动化调度

建议下一阶段优先事项：

1) 引入接口缓存与指标监控  
2) 增加模型离线评估报告自动生成  
3) 完善 E2E 联调与回归测试集
